## ⭐ ГЛАВНОЕ ОТКРЫТИЕ (2026-07-03, fork-replay) — ЦЕЛЕВОЙ РЫНОК ПЕРМИШЕНОВАН

**Рынок `0x322e1d53` (USDC/wS) ЗАКРЫТ вайтлистом ликвидаторов. Свежий бот НЕ МОЖЕТ его ликвидировать — на уровне протокола, не скорости.** Установлено fork-replay реальной ликвидации (блок 75048162) против ЖИВЫХ контрактов Silo:

- Изъятие залога идёт через `afterTokenTransfer`-гейт на **ЗАЛОГОВОМ** wS-share-токене; его гейдж `0x5c10d4cc71d2f66023ad93541bd1a773f92c3c07` — `PermissionedLiquidationController`. `allowMeToLiquidate()` закрыт ролью `ALLOWED_ROLE` (`0xd5dc6b38…`).
- **Вайтлист — ровно 4 адреса** (enumerable, прочитаны с гейджа): `0x0094c5d6`, `0x08a52ec3`, `0xb49329bf`, `0xccd487e0`. Оба «инкумбента» трека (`0xccd487` нового таргета И `0x0094c5` старого README) — в этом списке. Role-admin (DEFAULT_ADMIN) — единственный адрес `0x7461d8c0` (контракт, codesize 171).
- Свежий контракт → `OnlyAllowedRole()` при arm (проглатывается нашим try/catch) → `LiquidationNotAllowed()` при изъятии. Доказано тестом `test_Replay_PermissionedMarket_BlocksOutsider` (PASS).
- **Доминирование инкумбента (~78%) — это ГЕЙТ ДОСТУПА, не только скорость.** «Конкуренция», которую мерил трек (winner_probe: победитель перебивал по tip), — это конкуренция ВНУТРИ вайтлиста из 4, не открытый рынок. Прежний вывод «ставка решает» валиден только среди уже-впущенных.

**Почему трек это упустил (важно, чтобы не повторить):** гейт сидит на гейдже ЗАЛОГОВОГО силоса (`0xf55902de`→`0x5c10d4cc`), а всё индексировалось под ДОЛГОВЫМ (`0x322e1d53`→другой гейдж `0x36177720`). Проверка со стороны долга гейт не видит — **та же готча двусторонних пар, что кусала весь трек** (§2). `_turnOnLiquidation` взводит именно ЗАЛОГОВЫЕ share-токены — поэтому решает ЗАЛОГОВЫЙ гейт. Тул для воспроизведения на любом рынке: `analysis/permission_gate_check.py`.

**Контракт при этом ИСПРАВЕН и ПРИБЫЛЕН — блокирует только доступ.** Тест `test_Replay_DirectPool_WhenWhitelisted` (PASS): выдать нашему контракту `ALLOWED_ROLE` (cheatcode, как сделал бы вайтлистинг) → пол ревертит при `minProfit=max` И проходит при `minProfit=1`, реализуя **10.155405 USDC** на реальной ликвидации через прямой своп в пуле `0x324963c2` (route-A, всё ончейн, без агрегатора). Это **чуть больше**, чем инкумбент выжал сплит-маршрутом (~10.03 USDC) — тезис «чистый route-A structurally лучше» подтверждён числом.

**Кто такой админ `0x7461d8c0` — ВЫЯСНЕНО (2026-07-03): это Silo protocol governance, НЕ инкумбент.** Доказательства (ончейн, авторитетно):
- **Владеет `SiloFactory` `0xa42001d6`** — контрактом, который деплоит ВСЕ рынки Silo на Sonic. Владелец фабрики = governance протокола по определению; инкумбент-ликвидатор ей владеть не может.
- Владеет `GaugeHookReceiver` и `DEFAULT_ADMIN` гейджей МНОЖЕСТВА рынков — и gated `0x322e1d53`, И ОТКРЫТОГО `0x7e88ae5e` (его hook `0xddba7138` тоже owned этим Safe). Приватный мультисиг инкумбента не владел бы инфраструктурой открытого рынка, где чужие боты свободно конкурируют.
- Был initial owner/деплойером **xSILO** (`0x44517657…`, гов-токен Silo; ныне ownership у downstream Silo-DAO `0xE8e8041c`).
- Это **Gnosis Safe 1.4.1, 3-of-4**, подписанты `0x1fF60e85`/`0xe153437b`/`0x66B416a3`/`0x22FF5BCE` — НИ ОДИН не совпадает с адресами пайплайна инкумбента (0xccd487/0xa2712025/0x016e1a57/…). Подписанты Safe ≠ операторы ботов.

**Следствие для whitelisting-пути:** контрагент — **Silo Labs/governance, публичная сторона**, не закрытый картель. Значит `ALLOWED_ROLE` в принципе получаем через обращение к Silo (не структурно закрыто). НО: Silo сознательно курировал ровно 4 ликвидатора на этом рынке (вероятно, связано с deployer-механикой Silo V2 — создатель рынка настраивает его и делит revenue). Путь = соцконтакт с Silo, не ончейн-действие; исход не гарантирован. Это НЕ zero-capital-ончейн-шаг — требует внешней коммуникации и, вероятно, доверия/репутации.

**Стратегическое следствие (решать пользователю):** на этом рынке две опции — (1) **получить `ALLOWED_ROLE`** через Silo governance (публичный контрагент, но курируемый доступ, офчейн-процесс), либо (2) **пивот на ОТКРЫТЫЕ рынки** — ровно то, для чего в треке уже есть `radar/radar.py` (искать молодые/негейтед lending-рынки). Вероятно, Sonic использует другую версию исходника оракула (у Silo core минимум 5 версий за историю), не критично после прямого чтения конфига выше, но не закрыто окончательно.

### Пайплайн конкурента — НЕ один бот, а координированная связка
Инкумбент `0xccd487e01e9df6932f656b53668f58005f604417` (~78% приза по объёму) — транзитное звено, не самостоятельный актор. Установлено построчным Transfer-log анализом (`net_margin.py --decompose`):

**Два маршрута:**
- **Маршрут A** (чистый, без потерь): флеш-кредитор `0x578ee1ca…`→`0xf307ef18…` → sender-транзит `0xccd487` (держит ровно 0) → роутер `0x157f2158…`→`0x8f10b468…` (10+ пулов) → **бенефициар `0x016e1a57…`** (системно подтверждён: +$4,116/мес, 49/121 tx, в плюсе по обоим токенам — не AMM-пул)
- **Маршрут B** (лоссовый): флеш-кредитор `0xa2712025…` → `0xccd487` (малый остаток) → роутер `0x3a5d6a7a…` (40+ пулов, фиксированный набор) → **теряет 2.3–4.2% залога сжиганием в `0x0`** (переменная доля, подтверждено на 3 блоках — исключает fee-on-transfer токена, это механика роутера/пула). Бенефициар маршрута B **не единый** — хвост дробится между несколькими адресами.
- Правило выбора A vs B неизвестно (не по размеру сделки, не по времени)

**Ключевой вывод для стройки:** маршрут A не теряет ничего на burn — если реализовать чистый своп (как A), это структурное преимущество над инкумбентовым маршрутом B, где он сам теряет 2-4%.

### Конкуренция — измерена, после чистки от шума связки
`winner_probe.py` изначально показывал видимость аукциона (32/52 «равно» по tip), но при исключении известных pipeline-адресов (те же 0xa2712025, 0x016e1a57 и т.д. — они и вызывающие, и получающие в собственных транзакциях победителя) картина схлопывается до **n=2 подлинно внешних блоков**. В обоих победитель **строго перебил** внешнего конкурента по tip — первый чистый сигнал за весь трек в пользу «ставка решает», а не «просто скорость». Мало данных (n=2), но не шум.

`check_owner.py`: `0xccd487`/`0xa2712025` оба ревертят на `owner()` — тест общего контроля неубедителен ни в одну сторону.

### Важная механика двусторонних пар Silo V2 (готча для любых будущих тулов!)
`getConfigsForSolvency(borrower)` (в `SiloConfig.sol`) резолвит долговую/залоговую сторону **по РЕАЛЬНОЙ позиции заёмщика** (`getDebtSilo` смотрит балансы debtShareToken на ОБЕИХ сторонах пары), **не по тому, какой адрес силоса передан аргументом**. Значит `SiloLens.getUserLTV(silo_X, borrower)` может вернуть LTV заёмщика, чей реальный долг находится на **противоположной** стороне пары (занял wS под USDC-залог — зеркально нашему целевому направлению). Единственный способ узнать, что долг именно **на нужной** стороне — независимая проверка `debtBalanceOfUnderlying(ИМЕННО_ЭТОТ_силос, borrower)` (= `maxRepay`, читает share-баланс конкретно этого силоса). Если она возвращает 0, а LTV/LT ненулевые — заёмщик **не в нашем рынке**, статус LTV нерелевантен. `open_positions.py` v4 это учитывает (см. §4).

**Механизм РАЗРЕШЁН (был открытым вопросом, закрыт замером).** Ни C1 (долг на другой стороне пары), ни C2 (позиции нет вообще) не подтвердились — оказался третий, более простой механизм. `maxRepayShares` (сырой `balanceOf` debt-токена, без конвертации) на обоих проверенных адресах показал **1 wei долей** на нашей же стороне пары; `maxRepay`/`debtBalanceOfUnderlying` (конвертированные в активы через `convertToAssets`) при такой микроскопической доле **колеблется между 0 и 1–2 raw-единицами от блока к блоку** (сдвиг `totalSiloAssets/totalShares` из-за непрерывного начисления процентов) — то есть сравнение `debt_raw == 0` **ненадёжно** на этой границе: один и тот же адрес может в разных прогонах показать то `0`, то `1`, то `2`. `getLtv` считает по сырым долям (не по округлённым активам), поэтому LTV остаётся большим (390%+) даже когда актив-конвертация округляется в ноль. **Практический вывод для любого будущего кода:** для гейта «есть ли реальная позиция» использовать **порог в долларах** (`open_positions.py` — $1, консервативно выше шума округления и газовых издержек), не точное сравнение с нулём; при необходимости дополнительно проверять сырые доли (`maxRepayShares`/`analysis/debt_shares.py`) для различения «доли есть, актив округлился в 0» (пыль) от «долей и правда нет» (нет позиции).

### wS/S = 1:1 подтверждено МАТЕМАТИЧЕСКИ (закрыто 2026-07-03, было «вероятно»)
Ревью верно требовало закрыть «wS вероятно 1:1» — это лежало под моделью прибыли. Закрыто прямым чтением:
- wS = `0x039e2fb66102314ce7b64ce5ce3e5183bc94ad38` (из живого резолва залогового токена пары). **Нативный S-баланс контракта == totalSupply ДО ПОСЛЕДНЕГО WEI** (diff=0 на 27-значных числах) на трёх блоках: текущем, `72462468` (replay-блок) и 40M. Селекторы `deposit()`/`withdraw(uint256)` в байткоде — WETH9-паттерн: каждый wS атомарно погашаем в ровно 1 S, депег на уровне контракта невозможен.
- Уточнение зоны риска: корректность ДЕТЕКЦИИ от пега не зависела вовсе (SiloLens исполняет ту же оракульную математику, что и протокол при решении о ликвидируемости — детектор согласован с протоколом по построению). Пег бьёт по МОДЕЛИ ПРИБЫЛИ (продажа изъятого wS) — и 1:1-редемпшн ограничивает этот риск DEX-спредом, уже покрытым `slippage_bps`.

### Оракул wS — deviation threshold и quote-токен (закрыто)
Агрегатор `0xc76dfb89…` — Chainlink-прокси (`aggregator()` → реальная имплементация `0x0bdbff19543b20d0bc2d1ea08dee2be4c0b76743`), фид называется **«S / USD»** — цена нативного Sonic (S), не wS-специфичная (wS = обёрнутый S, вероятно 1:1). Подтверждено на `data.chain.link` и ончейн-зондом:
- **Deviation threshold = 0.5%** (тир Medium/High Market Risk, как у прочих волатильных активов на Sonic — сравнимые стейблкоин-фиды держат 0.3%). Практическое следствие: при обычной крипто-волатильности S 0.5%-движение внутри суток — рядовое дело, значит агрегатор, вероятно, обновляется **чаще**, чем раз в 24ч (heartbeat) — событий для live-триггера больше, чем «раз в сутки».
- **Quote-токен `0xad525f34…`** — НЕ ERC20-токен: 116 байт кода, `symbol()` ревёртит, `decimals()`=8 (та же точность, что у самого ценового фида). Вывод: синтетический sentinel «USD» с зашитой точностью, не реальный контракт с балансом/переводами.
- **Незакрытая мелочь, не блокирует:** страница `data.chain.link` показывает Heartbeat как `23:41:07`, что НЕ совпадает с прямым ончейн-чтением `87001с` (24ч10м01с). Рабочая гипотеза — это живой обратный отсчёт до следующего апдейта (сайт также показывает «Last update... N minutes ago» относительным таймером), не статичный интервал; арифметика (`87001 − прошедшее время`) грубо сходится, но не проверено, что это тикающий счётчик — не факт, гипотеза.

### ⚠️ Критично: `install_X.sh` vs `update_X.sh` — для четырёх тулов нужны ОБА, по порядку

Для `latency_probe`/`permission_check`/`radar` оба файла оставлены — `update_*` не проверялся настолько же подробно на самодостаточность в этой сессии, безопаснее гнать оба по порядку.

### Отдельная стратегическая ветка — `radar/radar.py`

Не активная нить этого трека — не используется в текущей работе над `0x322e1d53`, но рабочий и валидный, если понадобится вернуться к стратегии «искать новые рынки» вместо «выжимать зрелый».

## 6. Методология трека (держать и дальше)

- **Не гадать, проверять.** Каждое подозрительное число — повод для независимой перепроверки другим путём (пример: `debtBalanceOfUnderlying` через `maxRepay` как кросс-чек `getUserLTV`), не для принятия на слово.
- **Хирургические правки.** Расширения существующих функций — аддитивные (новый опциональный параметр с дефолтом), не рефакторинг.
- **Установщики — только программная генерация.** `body = open("файл.py").read()` → сборка bash-скрипта → `py_compile` + **реальный импорт** (не только синтаксис — он не ловит битые imports) + юнит-тесты → байт-в-байт валидация в чистой директории → **буквальный `cat`** для передачи, никогда не перепечатывать руками.
- **Тесты установщиков — без сторонних пакетов.** `Crypto.Hash.keccak` доступен в песочнице разработки, но не на VPS (stdlib-only). Селекторы вычисляются offline один раз, дальше — литеральное сравнение строк в шипованном тесте.
- **Guard-проверки на конкретные символы, не только на существование файла.** Пример: `grep -q "topic0" contestation.py`, не просто `[ -f contestation.py ]` — файл может существовать, но быть старой версией.
- **Ссылки на «см. предыдущее сообщение» внутри bash-блоков запрещены.** Всё, что должно быть исполнено, — исполняемая команда в блоке; всё, что комментарий для человека, — вне bash-блока, в тексте ответа.

### Execution-слой: route-A билдер + латентность (2026-07-03) — компьют НЕ узкое место
Собран оффчейн-билдер route-A квоты (`route_a_builder.py`) + замерена end-to-end латентность (`latency_e2e.py`):
- **Route-A калдата строится за 0.004мс** (чистый локальный encode) — структурное преимущество над агрегаторным путём, где HTTP-квота на каждую возможность добавляет 50–300мс (часто рвёт бюджет в 1 блок). Направление пула — одно кэшируемое чтение. Adapter берёт amountIn из allowance, поэтому пере-quote'ить между детекцией и сабмишеном нечего.
- **Горячий путь (детекция+сборка+tx-prep) = 3 RPC-раундтрипа + доли мс компьюта.** Отсюда (публичный HTTPS-RPC, 93мс/RTT) = **282мс**; у ко-located бота (RTT <1мс) ≈ **3.3мс**. Влезает даже в бюджет lag=1 (~1000мс) с запасом.
- **Вывод: наш софт (билдер/детектор) — НЕ связывающее ограничение.** Связывают: (1) близость к RPC-ноде (публичный 93мс vs ко-located <1мс) и (2) при lag=1 — приоритет в секвенсере (tip + близость), не компьют. То есть оставшиеся конкурентные рычаги — ИНФРАСТРУКТУРА (своя нода рядом с секвенсером) и tip-стратегия, а не оптимизация кода.
- Не замерено (обозначено честно): `eth_estimateGas` (нужен задеплоенный контракт+роль; +1 RPC или прекомпьют — газ executeLiquidation стабилен), подпись (~0.2мс локально), broadcast (+1 RPC). Полный путь ≈ 4–5 RPC + 0.2мс; ко-located ≈ 5мс — всё ещё в бюджете.

**Что это даёт для развилки:** для МЯГКОГО открытого рынка (lag 2–3, `0x11238006`) детекция+исполнение уже достаточно быстры даже с непредставительного публичного RPC — go-live-барьер здесь не софт, а (а) funded key + (б) нода рядом с секвенсером Sonic для схлопывания RPC-RTT. Для ГОРЯЧИХ рынков (same-block) нужен ещё mempool-слой (бэкран оракула) — отдельный крупный трек.

**Отдельный трек, не в очереди сборки:** getRoundData на момент исторического блока для точной реализованной маржи пайплайна конкурента (маршрут A/B) — это анализ экономики инкумбента, не задача paper-детектора. Смешивать с очередью выше не стоит; вести отдельно, если/когда понадобится.
