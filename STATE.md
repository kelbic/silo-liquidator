## 2. Установленные факты (высокая уверенность — прямые чтения, не выводы)

- `liquidationTargetLtv` ≈ **78%** — куда партиал-ликвидация возвращает позицию

### Оракул — подтверждено НАПРЯМУЮ на обеих сторонах, не по совпадению адреса
И долговая (`0x688951ad…`), и залоговая (`0x5da3510d…`) стороны резолвятся в имплементацию `0x2e226f3a…` = **ChainlinkV3Oracle**. Подтверждение на уровне «прочитан конфиг агрегатора», не «похоже на тот же класс»:
- `oracleConfig()` → `getConfig()` дал: agregator `0xc76dfb89…`, **heartbeat = 87001с (~24.2ч)**, secondaryAggregator = `0x0` (единственный фид)
- Class-маркер `getAggregatorPrice(bool)` (селектор `0xc71ed1e6`) найден в байткоде имплементации — это конкретное имя функции из реального исходника `ChainlinkV3Oracle.sol`, не общий паттерн
- `base_token` конфига совпал побайтово с известным адресом wS
- **Вывод:** офчейн-рва (VAA/pull-механизма) нет ни с одной стороны. Момент ликвидируемости целиком виден из ончейн-стейта — это разблокирует paper-детекцию по цене без внешних фидов.
- **Открытый мелкий вопрос:** попытка байт-в-байт сверить локальную компиляцию (из Arbitrum-артефакта Silo) с реально развёрнутым кодом разошлась на байте 112 — не в хвосте метаданных. Вероятно, Sonic использует другую версию исходника оракула (у Silo core минимум 5 версий за историю), не критично после прямого чтения конфига выше, но не закрыто окончательно.

### Пайплайн конкурента — НЕ один бот, а координированная связка
Инкумбент `0xccd487e01e9df6932f656b53668f58005f604417` (~78% приза по объёму) — транзитное звено, не самостоятельный актор. Установлено построчным Transfer-log анализом (`net_margin.py --decompose`):

**Два маршрута:**
- **Маршрут A** (чистый, без потерь): флеш-кредитор `0x578ee1ca…`→`0xf307ef18…` → sender-транзит `0xccd487` (держит ровно 0) → роутер `0x157f2158…`→`0x8f10b468…` (10+ пулов) → **бенефициар `0x016e1a57…`** (системно подтверждён: +$4,116/мес, 49/121 tx, в плюсе по обоим токенам — не AMM-пул)
- **Маршрут B** (лоссовый): флеш-кредитор `0xa2712025…` → `0xccd487` (малый остаток) → роутер `0x3a5d6a7a…` (40+ пулов, фиксированный набор) → **теряет 2.3–4.2% залога сжиганием в `0x0`** (переменная доля, подтверждено на 3 блоках — исключает fee-on-transfer токена, это механика роутера/пула). Бенефициар маршрута B **не единый** — хвост дробится между несколькими адресами.
- Правило выбора A vs B неизвестно (не по размеру сделки, не по времени)

**Ключевой вывод для стройки:** маршрут A не теряет ничего на burn — если реализовать чистый своп (как A), это структурное преимущество над инкумбентовым маршрутом B, где он сам теряет 2-4%.

### Конкуренция — измерена, после чистки от шума связки
`winner_probe.py` изначально показывал видимость аукциона (32/52 «равно» по tip), но при исключении известных pipeline-адресов (те же 0xa2712025, 0x016e1a57 и т.д. — они и вызывающие, и получающие в собственных транзакциях победителя) картина схлопывается до **n=2 подлинно внешних блоков**. В обоих победитель **строго перебил** внешнего конкурента по tip — первый чистый сигнал за весь трек в пользу «ставка решает», а не «просто скорость». Мало данных (n=2), но не шум.

`check_owner.py`: `0xccd487`/`0xa2712025` оба ревертят на `owner()` — тест общего контроля неубедителен ни в одну сторону.

### Важная механика двусторонних пар Silo V2 (готча для любых будущих тулов!)
`getConfigsForSolvency(borrower)` (в `SiloConfig.sol`) резолвит долговую/залоговую сторону **по РЕАЛЬНОЙ позиции заёмщика** (`getDebtSilo` смотрит балансы debtShareToken на ОБЕИХ сторонах пары), **не по тому, какой адрес силоса передан аргументом**. Значит `SiloLens.getUserLTV(silo_X, borrower)` может вернуть LTV заёмщика, чей реальный долг находится на **противоположной** стороне пары (занял wS под USDC-залог — зеркально нашему целевому направлению). Единственный способ узнать, что долг именно **на нужной** стороне — независимая проверка `debtBalanceOfUnderlying(ИМЕННО_ЭТОТ_силос, borrower)` (= `maxRepay`, читает share-баланс конкретно этого силоса). Если она возвращает 0, а LTV/LT ненулевые — заёмщик **не в нашем рынке**, статус LTV нерелевантен. `open_positions.py` v4 это учитывает (см. §4).

**Механизм РАЗРЕШЁН (был открытым вопросом, закрыт замером).** Ни C1 (долг на другой стороне пары), ни C2 (позиции нет вообще) не подтвердились — оказался третий, более простой механизм. `maxRepayShares` (сырой `balanceOf` debt-токена, без конвертации) на обоих проверенных адресах показал **1 wei долей** на нашей же стороне пары; `maxRepay`/`debtBalanceOfUnderlying` (конвертированные в активы через `convertToAssets`) при такой микроскопической доле **колеблется между 0 и 1–2 raw-единицами от блока к блоку** (сдвиг `totalSiloAssets/totalShares` из-за непрерывного начисления процентов) — то есть сравнение `debt_raw == 0` **ненадёжно** на этой границе: один и тот же адрес может в разных прогонах показать то `0`, то `1`, то `2`. `getLtv` считает по сырым долям (не по округлённым активам), поэтому LTV остаётся большим (390%+) даже когда актив-конвертация округляется в ноль. **Практический вывод для любого будущего кода:** для гейта «есть ли реальная позиция» использовать **порог в долларах** (`open_positions.py` — $1, консервативно выше шума округления и газовых издержек), не точное сравнение с нулём; при необходимости дополнительно проверять сырые доли (`maxRepayShares`/`analysis/debt_shares.py`) для различения «доли есть, актив округлился в 0» (пыль) от «долей и правда нет» (нет позиции).

### Оракул wS — deviation threshold и quote-токен (закрыто)
Агрегатор `0xc76dfb89…` — Chainlink-прокси (`aggregator()` → реальная имплементация `0x0bdbff19543b20d0bc2d1ea08dee2be4c0b76743`), фид называется **«S / USD»** — цена нативного Sonic (S), не wS-специфичная (wS = обёрнутый S, вероятно 1:1). Подтверждено на `data.chain.link` и ончейн-зондом:
- **Deviation threshold = 0.5%** (тир Medium/High Market Risk, как у прочих волатильных активов на Sonic — сравнимые стейблкоин-фиды держат 0.3%). Практическое следствие: при обычной крипто-волатильности S 0.5%-движение внутри суток — рядовое дело, значит агрегатор, вероятно, обновляется **чаще**, чем раз в 24ч (heartbeat) — событий для live-триггера больше, чем «раз в сутки».
- **Quote-токен `0xad525f34…`** — НЕ ERC20-токен: 116 байт кода, `symbol()` ревёртит, `decimals()`=8 (та же точность, что у самого ценового фида). Вывод: синтетический sentinel «USD» с зашитой точностью, не реальный контракт с балансом/переводами.
- **Незакрытая мелочь, не блокирует:** страница `data.chain.link` показывает Heartbeat как `23:41:07`, что НЕ совпадает с прямым ончейн-чтением `87001с` (24ч10м01с). Рабочая гипотеза — это живой обратный отсчёт до следующего апдейта (сайт также показывает «Last update... N minutes ago» относительным таймером), не статичный интервал; арифметика (`87001 − прошедшее время`) грубо сходится, но не проверено, что это тикающий счётчик — не факт, гипотеза.

### ⚠️ Критично: `install_X.sh` vs `update_X.sh` — для четырёх тулов нужны ОБА, по порядку
Для `contestation.py`, `latency_probe.py`, `permission_check.py` и `radar/radar.py` в репозитории есть **обе** пары файлов. `install_X.sh` — это база (более старая версия), `update_X.sh` — патч поверх неё (добавляет поле/параметр). Проверено напрямую: `install_contestation.sh` не содержит параметр `topic0` (нужен `open_positions.py`), `install_latency_probe.sh` не содержит поле `idx` (нужен `winner_probe.py`). Запускать **в порядке `install_X.sh` → `update_X.sh`**, никогда только первый. `install_contestation.sh` был настолько устаревшим, что исключён из этого репо целиком — держи только `update_contestation.sh` для этого файла (он самодостаточен, содержит полный файл, не патч). Для `latency_probe`/`permission_check`/`radar` оба файла оставлены — `update_*` не проверялся настолько же подробно на самодостаточность в этой сессии, безопаснее гнать оба по порядку.

### Отдельная стратегическая ветка — `radar/radar.py`
Из более ранней фазы трека, другой угол атаки: не контестировать зрелый рынок (обойма co-located профи уже осела), а находить **молодые** lending-рынки, где хвост ликвидаций ещё открыт (аналогия — Sonic в январе-2025, пока searcher'ы не подтянулись). Свой systemd-таймер, своя SQLite (`radar.db`), алерты в Telegram напрямую через `api.telegram.org` — **явно спроектирован не трогать общий venv/инфраструктуру ботов** (`--telegram-token`/`--chat-id` как CLI-аргументы, не хардкод чужого бота). Ноль капитала, разовый запуск, не демон. Не активная нить этого трека — не используется в текущей работе над `0x322e1d53`, но рабочий и валидный, если понадобится вернуться к стратегии «искать новые рынки» вместо «выжимать зрелый».

## 6. Методология трека (держать и дальше)

- **Не гадать, проверять.** Каждое подозрительное число — повод для независимой перепроверки другим путём (пример: `debtBalanceOfUnderlying` через `maxRepay` как кросс-чек `getUserLTV`), не для принятия на слово.
- **Хирургические правки.** Расширения существующих функций — аддитивные (новый опциональный параметр с дефолтом), не рефакторинг.
- **Установщики — только программная генерация.** `body = open("файл.py").read()` → сборка bash-скрипта → `py_compile` + **реальный импорт** (не только синтаксис — он не ловит битые imports) + юнит-тесты → байт-в-байт валидация в чистой директории → **буквальный `cat`** для передачи, никогда не перепечатывать руками.
- **Тесты установщиков — без сторонних пакетов.** `Crypto.Hash.keccak` доступен в песочнице разработки, но не на VPS (stdlib-only). Селекторы вычисляются offline один раз, дальше — литеральное сравнение строк в шипованном тесте.
- **Guard-проверки на конкретные символы, не только на существование файла.** Пример: `grep -q "topic0" contestation.py`, не просто `[ -f contestation.py ]` — файл может существовать, но быть старой версией.
- **Ссылки на «см. предыдущее сообщение» внутри bash-блоков запрещены.** Всё, что должно быть исполнено, — исполняемая команда в блоке; всё, что комментарий для человека, — вне bash-блока, в тексте ответа.

## 7. Приоритетные следующие шаги

Оба дешёвых ончейн-вопроса закрыты (см. §2, «Оракул wS — deviation threshold и quote-токен»). Остаётся один инженерный кусок сборки:

1. **[Средне]** Полный охват заёмщиков через `debtShareToken` Transfer-историю (держатели с `balanceOf>0`) — снимает окно-ограничение `open_positions.py`, единственное известное на сейчас.
2. **[Долгосрочно]** Live-детектор поверх `open_positions.py`: триггер на обновление агрегатора `0xc76dfb89` — deviation threshold 0.5% на волатильном S означает обновления, вероятно, ЧАЩЕ, чем раз в 24ч (heartbeat) — 0.5%-движение для крипто-актива в течение суток обычное дело. Это хорошо для стратегии: событий для реакции больше, чем «раз в сутки».

**Отдельный трек, не в очереди сборки:** getRoundData на момент исторического блока для точной реализованной маржи пайплайна конкурента (маршрут A/B) — это анализ экономики инкумбента, не задача paper-детектора. Смешивать с очередью выше не стоит; вести отдельно, если/когда понадобится.
